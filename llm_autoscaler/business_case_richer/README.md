# business_case_richer/

Real-cluster business-case figures from `results_richer/` (the N=3-replicated,
two-trace sweep: `cpu_bursty` + `wiki_diurnal`). Generated via
`plot_business_case_real.py` and `plot_percentiles_real.py` against
`results_richer/{cpu_bursty,wiki_diurnal}`.

**Differs from `../business_case_real/`**, which holds the earlier
`results_k8s_v2` figures (single rep, no `gpt-oss-120b`, no wiki_diurnal
breakout). Do not conflate the two — file naming here always ends in
`_bursty` or `_wiki` to make the source trace explicit; the old directory's
files end in plain `_cpu`/`_io`.

Snapshot as of 2026-07-17 09:20 UTC: cpu_bursty is 8/9 controllers at full
N=3 (Qwen3-80B partial, being retired); wiki_diurnal is partial (see
`summary_wiki_diurnal.csv` / `percentiles_wiki_diurnal.csv` for exact reps
completed per config). Note: GPT-OSS-120B's `cot` variant on cpu_bursty
shows a reproducible scale-oscillation bug (P99 ~3.8s, 63% success) across
all 3 reps — visible as the extreme outlier in the ladder/tail-ratio charts.

## `ablation_llm_overhead.{csv,png}` — API overhead vs. local hosting

Generated by `../ablation_llm_overhead.py`. Answers: how much of each LLM
controller's decision latency is provider/network overhead that a local (or
better) deployment would remove, versus latency that's really the model's
own compute? We have not run these models locally — this decomposes the
*already-measured* `llm_latency_ms` column (cpu_bursty) using two facts per
model instead of guessing:

- `median_s` / `mean_s` / `p90_s` — median is the "typical, nothing-went-wrong"
  call; the gap to mean/p90 is retry/timeout inflation.
- `step_error_rate_pct` / `logged_rotations` — actual HTTP 504/503 counts and
  provider-fallback rotations grepped from `logs/richer_auto_cpu_bursty_*.log`
  (ground truth for *why* the tail exists, not inferred from latency alone).

The added column, **`local_hosting_expected_effect`**, is the qualitative
call this evidence supports:

| model | median decision | step error rate | local-hosting verdict |
|---|---|---|---|
| Mistral-Small / Llama-8B | ~0.5s | 0% | Minimal — already near the floor |
| GPT-OSS-120B | 0.6s (median), 14.3s (mean) | 0% errors, 31 rotations/264 steps | Moderate — removes rate-limit fallback rotations; typical case barely changes |
| Llama-70B | 26.0s (43% of the 60s interval) | 1.1% | Partial — removes queueing/network RTT, but genuine 70B compute remains |
| Qwen3-80B | 80.4s (134% of the interval) | 80.3% | Large — endpoint is effectively dead; almost all overhead is provider-side |

Qwen3-80B and Llama-70B are qualitatively different failure modes despite
both having "slow" means: Qwen's overhead is ~entirely a broken endpoint
(504s), so local hosting (or literally any healthier provider) should
recover most of it. Llama-70B's overhead is real even at the median with a
near-zero error rate, so it's closer to genuine model compute — local
hosting would likely still help (no multi-tenant queueing) but won't be
free.
